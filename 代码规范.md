# 代码规范

> jgeted + T 共用。遵守它，两个人写的代码看起来像一个人写的。

---

## 一、语言约定

| 场景 | 用什么 | 说明 |
|---|---|---|
| 类名、方法名、变量名 | 英文 | `NpcWorker`、`getResearchPoints()`、`maxPopulation` |
| 注释 | **中文** | 你是中国人，你用中文注释。方便彼此和未来的自己看懂 |
| Git 提交信息 | 中文 | `添加 NPC 人口控制逻辑`，不用英文也不用"update code" |
| 配置文件 | 英文 key / 中文注释 | `# 研究点数每次获得的数量` |

## 二、命名规范

### 类名
- 大驼峰：`NpcWorker`、`ResearchTracker`、`GuiSkillTree`
- 见名知意：`ContainerNpcInventory` 对，`Container123` 错

### 方法名
- 小驼峰：`getResearchPoints()`、`spawnNpc()`、`isHostileTo()`
- 动词开头：`get` / `set` / `is` / `has` / `can` / `add` / `remove`

### 变量名
- 小驼峰：`maxHealth`、`ownerUuid`、`currentPoints`
- 不要缩写到看不懂：`pl`（是什么？player？plugin？placeholder？）→ 写 `player`
- 布尔变量加前缀：`isAlive`、`hasPermission`、`canAttack`
- 常量全大写：`MAX_NPC_COUNT`、`DEFAULT_RESEARCH_POINTS`、`MOD_ID`

### 包名
```
com.你的名字.模组名.模块名.功能
```

例如：
```
```
com.jgeted.sagadyssey.core.research
com.jgeted.sagadyssey.npc.entity
com.jgeted.sagadyssey.structure.template
```

## 三、代码结构

### 每个类文件的顺序（固定）

```java
// 1. 包声明
package com.jgeted.ancientwar.npc.entity;

// 2. 导入（按字母排序，不要用 * 导入）
import net.minecraft.world.entity.EntityType;
import net.neoforged.bus.api.IEventBus;

// 3. 类声明 + 类注释
/**
 * NPC 工人的实体类。
 * 负责采集资源和搬运物品。可以通过给物品切换职业。
 */
public class NpcWorker extends NpcBase {
    
    // 4. 常量
    public static final int MAX_WORK_RANGE = 16;
    
    // 5. 字段
    private int workCooldown;
    
    // 6. 构造函数
    public NpcWorker(EntityType<?> type, Level level) {
        super(type, level);
    }
    
    // 7. 公共方法
    public void startWorking() { ... }
    
    // 8. 私有方法
    private boolean findResource() { ... }
}
```

### 缩进
- 用 **4 个空格**，不要用 Tab
- IntelliJ IDEA 里：Settings → Editor → Code Style → Java → Tabs and Indents → 勾选 "Use tab character" 的**反选**

### 大括号
```java
// 对——左括号在同一行
if (condition) {
    doSomething();
}

// 错——不要另起一行
if (condition)
{
    doSomething();
}
```

### 空行
- 方法之间空一行
- 逻辑段落之间空一行
- 不要太密也不要太松

## 四、Claude Code 协作规范

### 每次让 Claude 写代码时

```
请帮我实现 [具体功能]。要求：
1. 遵守项目代码规范（见 代码规范.md）
2. 中文注释
3. 写完后解释每一段做了什么
```

### Claude 写完后

- **必须读懂再提交。** 不要因为 Claude 写了就以为没问题——它是辅助，你是决策者
- **改了哪里，口头跟对方说一声。** 比如 "我把 NpcBase 的构造函数改了，你的 NpcWorker 继承应该不受影响"

### 提交到 Git 之前

- 先跑一次 `./gradlew build`，确保编译通过
- 如果跟 NPC 或实体有关，跑一次 `./gradlew runClient` 进游戏看一眼

## 六、Git 规范

### 提交粒度
- 一个功能点一次提交
- 不要攒了三天代码一起交

### 提交信息格式
```
动词 + 对象 + 补充说明

例：
添加 NPC 职业切换 GUI
修复营地在 null biome 时崩溃的问题
重构 NpcBase 构造函数，提取公共参数到父类
```

### 分支
- `main`：稳定可运行的
- `dev`：日常开发
- 大功能可以开 feature 分支：`feature/npc-spawn-system`

## 七、模组独有的约定

### MOD_ID 格式
- 小写字母 + 下划线：`ancient_war`（不是 `AncientWar` 或 `ancientwar`）
- 所有资源路径（纹理、模型、语言文件）用 MOD_ID 做文件夹名

### 注册命名
- 方块：`MOD_ID:campfire_block`
- 物品：`MOD_ID:blueprint_item`
- 实体：`MOD_ID:npc_worker`

### 日志
- 不要用 `System.out.println()`
- 用模组的 Logger：
```java
private static final Logger LOGGER = LogUtils.getLogger();
LOGGER.info("NPC 生成成功，位置：{}", position);
LOGGER.warn("找不到合适的营地位置");
LOGGER.error("NPC 数据加载失败", exception);
```

## 八、两人协作底线

1. **每天开始工作前**：看看对方昨天改了哪些文件（`git log --since=yesterday`）
2. **改了同一个文件**：先沟通，谁先改完谁先提交，另一个人拉取后合并
3. **有拿不准的**：写注释标记，不要默默删除对方的代码
4. **功能做完**：跑一遍 `runClient`，进游戏测过再提交
